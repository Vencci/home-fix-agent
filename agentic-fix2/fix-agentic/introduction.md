# Home Fix Agent — Project Introduction

## 1. What This Service Does

The Home Fix Agent is an AI-powered assistant that turns a photo of a broken household item into a purchase-ready product recommendation. A user takes a picture of a burned-out light bulb, a dead battery, a cracked outlet cover, or a damaged roof shingle. The system identifies the item, extracts the technical specifications needed to find a replacement (bulb base type, wattage, color temperature, battery size, filter dimensions — whatever is relevant), searches for matching products online, ranks them by fit and value, and presents the top options. If the user wants to buy, the system places the order — but only after the user explicitly confirms.

The problem it solves is a knowledge gap. The information needed to buy the correct replacement — E26 vs. E12 base, CR2032 vs. CR2025 battery, 20×25×1 vs. 20×25×4 filter — is usually printed on the broken item or visible in its surroundings, but most people cannot interpret it or translate it into the right search query. They end up guessing at the hardware store, buying the wrong part, and making a second trip. This system eliminates that friction: the photo is the input, the correct product is the output.

The service is human-in-the-loop by design. It never places an order without the user reviewing the product, price, and quantity and explicitly confirming. It is built for common, low-complexity household items — not complex renovations or professional contracting work. The target users are homeowners without hardware expertise, renters who need to replace items quickly, and property managers handling maintenance across multiple units.

The interaction is conversational, not transactional. When the system is uncertain about a specification — say, whether a bulb is on a dimmer switch — it asks the user before searching. The pipeline blocks at this clarification step: no products are shown until the user answers. After products are presented, the user can refine the results by typing follow-up messages like "I need dimmable" or "show me cheaper options," and the system re-searches without requiring a new photo upload. The full conversation — every question, answer, and system message — is stored as part of the session transcript.

The system is deployed as a web application on Fly.io and is also available as a CLI tool. The web interface is a chat-driven UI: users upload a photo, see the analysis and specs appear as chat messages, answer clarification questions in the same chat thread, browse product cards inline, and place an order with a single click. A sidebar shows past sessions that can be loaded to review the full conversation. The CLI provides the same interactive flow through terminal prompts. Both interfaces share the same incremental pipeline engine, ensuring consistent behavior.


## 2. How It Was Built

The system was developed using a spec-driven approach. A detailed specification document was written first, defining the problem statement, functional requirements, data models, agent responsibilities, JSON contracts, test plan, and acceptance criteria — before any code was written. The implementation then followed the spec module by module, with each component built to satisfy its specified inputs, outputs, and acceptance criteria.

The technology stack is Python with Pydantic for data models, OpenAI's GPT-4o for multimodal vision and text reasoning, SerpAPI for Google Shopping product search, FastAPI for the web server, and Jinja2 for report rendering. The system stores all session artifacts as JSON files on disk — no database is required for the MVP. Deployment uses Docker on Fly.io, with API keys managed as encrypted secrets.

The implementation is structured as an incremental pipeline with five agent modules and a pipeline engine (`pipeline.py`) that advances through stages one at a time. Each agent is a plain Python function with a strict input/output contract. The pipeline engine exposes two functions — `start_session` and `advance` — that both the CLI and web UI call, ensuring identical behavior across interfaces.

1. **Photo Intake** validates the uploaded image (format, size) and copies it to a session directory.
2. **Vision Analyst** sends the photo to GPT-4o with a constrained prompt and receives a structured JSON analysis: item category, problem type, visible text, physical description, and a confidence score.
3. **Spec Extractor** takes the analysis and the photo, and produces a full product specification with per-field confidence scores and a search query string. When confidence on a critical field is below 0.7, it generates clarification questions for the user. **The pipeline blocks here** — it does not proceed to product search until the user answers. The user's response is passed back to the Spec Extractor as additional context, and it re-runs to produce updated specs.
4. **Product Searcher** queries for matching products using a three-tier strategy: SerpAPI (real Google Shopping results) if an API key is configured, curated mock data for known categories (bulbs, batteries, filters, hardware), or LLM-generated product suggestions as a final fallback for categories not covered by the mock data.
5. **Product Ranker** scores results using a weighted formula — 40% specification match, 20% customer rating, 20% price, 10% availability, 10% review count — and returns the top five with a human-readable explanation for each recommendation.

After ranking, the user can either select a product to order or send a refinement message (e.g., "I need dimmable", "show me cheaper options"). Refinement appends the user's feedback to the search query and re-runs the search and ranking cycle without re-analyzing the photo. The **Order Manager** presents a confirmation summary when the user selects a product. Only an explicit "yes" (CLI) or a confirmed dialog (web UI) triggers order placement. The order is recorded regardless of whether the user confirms or cancels, so every interaction is auditable.

Every step of the interaction is recorded as a `ChatMessage` in the session's `messages` list, with a role (user, assistant, or system), content, timestamp, and the pipeline stage at which it was created. This conversation transcript is persisted as part of the session JSON and can be replayed in full.

The web UI is a chat-driven single-page application served by FastAPI. It presents the pipeline as a conversation: the user uploads a photo, the system's analysis appears as chat bubbles, extracted specs are shown as an embedded table card, clarification questions appear as assistant messages with the input bar prompting "Answer the questions above...", and products appear as embedded cards with Buy buttons. A sidebar lists past sessions with stage badges (clarifying, results, done, error) and clicking one loads the full conversation. The entire frontend is a single HTML file with inline CSS and JavaScript — no build step, no framework, no dependencies.

Several bugs were discovered and fixed during live testing. The initial deployment returned "Photo analysis failed" for every request because the error message from the LLM client (an authentication failure) was being swallowed and replaced with a generic message — the fix was to propagate the actual error. The Docker image excluded the mock product data because `.dockerignore` excluded the `tests/` directory where the data originally lived — the fix was to move the mock data to `data/`. The mock search returned LED bulbs for every query (including a broken roof) because the fallback returned the first category in the file regardless of relevance — the fix was to return empty when no category matches and fall through to LLM-generated results. Each of these bugs was caught by real user testing and fixed within minutes, demonstrating the value of a simple, debuggable architecture.


## 3. Why It Is Agentic

The system is agentic because it decomposes a complex, multi-step task — going from a photo of a broken item to a confirmed product order — into specialized reasoning steps, each powered by an LLM call with a distinct role and constrained behavior. This is not a single prompt that does everything. Each agent has a specific job, a specific prompt, and a specific output schema. The Vision Analyst is forbidden from recommending products. The Spec Extractor is forbidden from placing orders. The Order Manager cannot proceed without human confirmation. These constraints are enforced by the architecture, not by hoping the LLM follows instructions.

The agentic decomposition provides three concrete benefits over a monolithic approach:

**Separation of perception from action.** The agents that understand the photo (Vision Analyst, Spec Extractor) are completely separated from the agents that take action in the world (Product Searcher, Order Manager). This separation is a safety requirement. When real money is involved, the system must be transparent about what it observed, what it inferred, and what it is uncertain about — and it must stop and ask before spending. A monolithic system that goes from photo to purchase in a single LLM call would be opaque and dangerous.

**Debuggability through structured intermediate outputs.** Every agent produces a typed JSON record that is stored as part of the session. When a recommendation is wrong, you can inspect the chain: Did the Vision Analyst misidentify the item? Did the Spec Extractor get the wrong base type? Did the search query miss a key term? Did the ranker score a bad product too high? In a monolithic system, you would only see the final output and have no way to diagnose which reasoning step failed. The agentic structure makes every intermediate decision visible and auditable.

**Independent improvement of each reasoning step.** The Vision Analyst prompt can be improved without touching the ranking logic. The ranking weights can be tuned without changing the search query generation. The product search can be swapped from mock data to SerpAPI to Amazon's API without affecting any other agent. Each agent is a module with a contract — as long as the contract is satisfied, the internals can change freely. This modularity is what makes the system evolvable.

A deliberate architectural choice was to implement agents as plain Python functions rather than using a multi-agent orchestration framework. The agents do not communicate with each other, negotiate, or make autonomous decisions about what to do next. The pipeline runner calls them in a fixed sequence. This is simpler, cheaper, and more predictable than a framework where agents decide their own control flow. The "agentic" quality comes from the LLM-powered reasoning at each step and the separation of concerns between steps — not from autonomous agent behavior. For a system that handles real purchases, predictability is more valuable than autonomy.

The human-in-the-loop design is enforced at three architectural boundaries. First, the Spec Extractor's **blocking clarification gate** pauses the entire pipeline when confidence is low — the system will not search for products until the user has answered. This is not a soft suggestion; the pipeline stage is set to `CLARIFYING` and no downstream agent runs until the user responds via `/chat` or the CLI prompt. This prevents the common failure mode where a wrong spec leads to a confidently wrong recommendation. Second, the **refinement loop** at the `RESULTS` stage lets the user correct the system's output without starting over — typing "I need dimmable" re-runs the search with updated criteria, keeping the original analysis and photo intact. Third, the Order Manager's confirmation gate is a hard state machine boundary: the transition from `RESULTS` to `DONE` requires an explicit user action. There is no code path, no timeout, and no default that bypasses this gate.


## 4. How It Is Evaluated

Evaluating an agentic system that depends on non-deterministic LLM calls and live product search APIs requires a layered testing strategy that separates what can be verified automatically from what requires human judgment.

**Automated unit tests (26 tests, all passing)** cover the deterministic components. Schema validation tests verify that all Pydantic models instantiate, serialize, and deserialize correctly — including the new `PipelineStage`, `ChatMessage`, and `PipelineResult` with its conversation transcript. Photo intake tests confirm that the system rejects invalid formats (BMP), oversized files (>20 MB), and missing files, while accepting valid JPEGs. Ranking engine tests verify that the weighted scoring formula is deterministic, that higher spec match produces higher rank, that normalization handles edge cases, and that rank numbers are assigned sequentially. Order manager tests confirm that the confirmation gate works: creating an order sets status to PENDING, confirming sets it to PLACED with a timestamp and retailer ID, cancelling sets it to CANCELLED, and quantity multiplication is correct. Mock search tests verify that the fixture data loads and returns properly typed objects. Pipeline stage tests verify the interactive flow: `start_session` blocks at `CLARIFYING` when the spec extractor returns questions and proceeds to `RESULTS` when it does not; `advance` from `CLARIFYING` re-runs spec extraction with user context and moves to `RESULTS`; `advance` from `RESULTS` with refinement feedback re-searches and returns updated products. These tests run in under a second with no API keys or network access.

**LLM contract tests** verify the behavioral constraints of the vision and spec extraction agents. The Vision Analyst must return valid JSON matching the IssueAnalysis schema, must not recommend products, and must set confidence low for unclear images. The Spec Extractor must return valid JSON matching the ProductSpec schema, must generate clarification questions when confidence is below threshold, and must produce a non-empty search query. JSON schema compliance is checked programmatically; behavioral constraints are verified by human review on a set of test photos.

**Live user testing** caught three bugs that automated tests could not: the swallowed API error that produced a misleading "try a clearer photo" message, the Docker packaging issue that excluded mock data from the deployed image, and the blind mock fallback that returned LED bulbs for every unrecognized query. Each bug was found within minutes of deployment, diagnosed by reading the error output and tracing the code path, and fixed with a targeted change. This experience validates the architectural choice of structured error propagation and inspectable intermediate outputs — when something goes wrong, the system tells you what actually happened rather than hiding behind a generic message.

**Session replay** is the foundation for ongoing evaluation. Every pipeline run stores the complete chain of artifacts: the original photo, the Vision Analyst's JSON output, the extracted specs with per-field confidence, the search results, the ranked products, the order record, and the full conversation transcript with every user message, system message, and assistant response timestamped and tagged with the pipeline stage at which it occurred. The `replay` command (CLI) or the session sidebar (web UI) lets you load any past session and inspect every step — including what clarification questions were asked, what the user answered, and whether refinement changed the results. This makes it possible to answer specific diagnostic questions: Was the spec extraction wrong? Did the clarification question help? Was the search query too narrow? Did the ranker overweight price at the expense of spec match? When a prompt or ranking weight is changed, past sessions can be re-evaluated to measure whether the change improved or degraded the output.

**The manual test matrix** defines five scenarios with explicit pass criteria: a burned-out A19 bulb with visible text (must extract E26 and 60W), a CR2032 battery in a remote (must extract correct size), an HVAC filter with printed dimensions (must match), a blurry ceiling light photo (must report low confidence), and an out-of-category item like a faucet or roof (must not crash or return unrelated products). This matrix is run after any change to the LLM prompts, the ranking weights, or the search fallback logic. It serves as a lightweight regression suite for the non-deterministic parts of the system that automated tests cannot cover.
